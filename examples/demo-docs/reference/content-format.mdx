Portico expects a content repository with this layout:

| File | Purpose |
|------|---------|
| `docs.json` | Site name, navigation groups, navbar |
| `index.mdx` | Home page |
| `{path}.mdx` | One page per nav entry (e.g. `guide/quickstart.mdx`) |
| `i18n/{locale}/…` | Optional translations (e.g. `i18n/es/guide/welcome.mdx`) |

## Supported MDX components

- `<Note>`, `<Warning>`, `<Info>`
- `<Steps>` / `<Step title="…">`
- `<Card>`, `<CardGroup>`, `<Icon>`
- `<Tabs>` / `<Tab>`, `<CodeGroup>`, `<Accordion>`, `<Frame>`

Fenced code blocks use Shiki syntax highlighting. ` ```mermaid ` blocks render as diagrams.

## SEO frontmatter

Optional fields on any `.mdx` page:

| Field | Purpose |
|-------|---------|
| `title` | Page title and `<title>` tag |
| `description` | Meta description (auto-generated from body if omitted) |
| `sidebarTitle` | Shorter label in the sidebar |
| `ogImage` | Per-page social preview image (URL or site-relative path) |
| `draft: true` | Hidden from nav, sitemap, search, and `llms.txt`; emits `noindex` |
| `noindex: true` | Excluded from sitemap/search; emits `noindex` but stays in nav |

Site-wide defaults live in `content/site.yaml`: `ogImage`, `verification` (Google Search Console, etc.).

The portal also emits JSON-LD (`TechArticle`, `BreadcrumbList`, `WebSite`) and generates a default Open Graph image at `/opengraph-image`.

## OpenAPI reference (optional)

API reference pages come from OpenAPI YAML files in Portico's `content/openapi/` directory,
configured in `content/site.yaml` — not from the docs content repo.
